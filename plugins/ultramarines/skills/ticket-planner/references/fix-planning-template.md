# Fix Planning Template

## Purpose

Use this template for the final planning output. Keep it concise, structured, and useful for approval before coding.

## Template

```markdown
Suggested title
- type: TICKET-ID | short summary

Suggested implementation summary title
- type: TICKET-ID | short summary

Ticket summary
- Ticket: TICKET-ID
- Type: bug | feature | chore | refactor | hotfix
- Status: ...
- Priority: ...
- Summary: ...
- Analysis baseline: ...

Problem to solve
- ...

Proposed approach A
- Strategy: ...
- Why it works: ...
- Scope shape: ...

Proposed approach B if relevant
- Strategy: ...
- Why it might be chosen instead: ...
- Tradeoff difference: ...

Why this approach is recommended
- ...

Likely affected modules / files / areas
- Entry points: ...
- Shared UI / components: ...
- State / hooks / stores: ...
- Services / API / mapping / validation: ...
- Platform or environment boundaries: ...

State handling to consider
- ...

API/data dependencies
- ...

Risks / tradeoffs
- ...

Edge cases
- ...

Suggested tests to prepare
- ...

Recommendation
- Proceed | Needs clarification first
- Reason: ...
```

## Writing Rules

- Prefer short bullets.
- Separate confirmed facts from planning assumptions.
- Avoid code-level detail.
- Keep the recommended approach explicit.
- Keep the recommendation decision visible at the end.
